# Build a Transformer for Name Generation

You know JavaScript. You know how to build UIs, wrangle APIs, and ship products. Now you want to understand how transformers actually work -- not from a blog post with hand-wavy diagrams, but by building one yourself, piece by piece, watching it learn to generate human names.

We will start with the dumbest possible model (counting character pairs) and progressively build up to a full transformer with self-attention. At every step, the model generates names so you can see it getting smarter. By the end, you will go from gibberish like `"mnba"` to plausible names like `"alexandra"`.

This follows the "spiral development" approach from Andrej Karpathy's makemore series: each model works end-to-end before we add complexity.

## Bigram Model

The simplest language model you can build: look at one character, predict the next. That is it. No context, no memory, no neural network. Just counting.

The idea is straightforward. We scan through every name and count how often each pair of characters appears together. If `'t'` is followed by `'h'` a thousand times, that pair gets a high count. Then we normalize the counts into probabilities.

What we are really doing here is defining P(next character | previous character) -- a conditional probability distribution. A language model, at its core, is just a machine that assigns probabilities to sequences. Given some context, it tells you: how likely is each possible next token? The bigram model is the simplest version of this idea: the "context" is just one character.

<CodeBlock title="bigram_counts.py" language="python">{`# Count character pairs (bigrams)
N = torch.zeros((vocab_size, vocab_size), dtype=torch.int32)

for w in train_words:
    chs = ['.'] + list(w) + ['.']
    for ch1, ch2 in zip(chs, chs[1:]):
        ix1 = stoi[ch1]
        ix2 = stoi[ch2]
        N[ix1, ix2] += 1

# Normalize rows to get probabilities
# Adding 1 for smoothing (avoids division by zero and log(0))
P = (N + 1).float()
P = P / P.sum(dim=1, keepdim=True)`}</CodeBlock>

<Callout type="info" title="Why add 1? Laplace smoothing">
  <p>Notice the <code>N + 1</code> before normalizing. This is called Laplace smoothing (or add-one smoothing). Without it, any character pair that never appeared in the training data gets a count of zero, which means a probability of zero. That might sound harmless, but it is catastrophic when we compute loss: <code>log(0) = negative infinity</code>. A single unseen bigram would make our entire evaluation blow up. By adding 1 to every count, we guarantee that every character pair has at least a tiny probability. It is a crude but effective way to say &quot;nothing is truly impossible, just unlikely.&quot;</p>
</Callout>

A few Python/PyTorch things to unpack here:

- `torch.zeros((27, 27))` creates a 27x27 matrix of zeros. Think of it as `new Array(27).fill(null).map(() => new Array(27).fill(0))`, but way more efficient.
- `zip(chs, chs[1:])` pairs each character with the next one. So for `"emma"`, the pairs are `('.','e'), ('e','m'), ('m','m'), ('m','a'), ('a','.')`.
- `P.sum(dim=1, keepdim=True)` sums each row. The `dim=1` means "sum across columns," and `keepdim=True` keeps the shape compatible for broadcasting (like how JS spreads values when array shapes differ).

g = torch.Generator().manual_seed(2147483647)

for _ in range(10):
    out = []
    ix = 0  # start token
    while True:
        p = P[ix]
        ix = torch.multinomial(p, num_samples=1, replacement=True, generator=g).item()
        if ix == 0:  # end token
            break
        out.append(itos[ix])
    print(''.join(out))`}</CodeBlock>

`torch.multinomial` is the key function here -- it randomly picks an index based on the probability distribution `p`. It is like rolling a weighted die. If `P('h' | 't')` is high, the model is more likely to pick `'h'` after seeing `'t'`.

Let us also compute the loss so we can compare models later. The standard metric is **negative log likelihood** (NLL). Lower is better.

But why log likelihood? Think about what it means to evaluate a model on a whole word. The probability of `"emma"` is P(e|.) times P(m|e) times P(m|m) times P(a|m) times P(.|a). We are multiplying probabilities together. The problem is that multiplying many small numbers together quickly leads to numerical underflow -- the result becomes so tiny that floating-point numbers cannot represent it. Taking the logarithm fixes this: log(a times b) = log(a) + log(b). Products become sums, which computers handle gracefully.

There is a deeper connection to information theory here. The negative log probability of an event, measured in nats (or bits if you use log base 2), is the "surprise" or "information content" of that event. A highly probable event (P close to 1) carries little surprise. A rare event (P close to 0) is very surprising. The average NLL across many examples is the cross-entropy between the true data distribution and the model -- and a perfect model would achieve NLL equal to the entropy of the true distribution (the irreducible uncertainty in the data). You literally cannot do better than that.

<CodeBlock title="bigram_loss.py" language="python">{`# Compute negative log likelihood loss
log_likelihood = 0.0
n = 0

for w in val_words:
    chs = ['.'] + list(w) + ['.']
    for ch1, ch2 in zip(chs, chs[1:]):
        ix1 = stoi[ch1]
        ix2 = stoi[ch2]
        prob = P[ix1, ix2]
        log_likelihood += torch.log(prob)
        n += 1

nll = -log_likelihood / n
print(f"Bigram NLL: {nll.item():.4f}")  # ~2.45`}</CodeBlock>

<Callout type="info" title="What is negative log likelihood?">
  <p>Imagine the model says <code>P('m' | 'e') = 0.15</code>. The log of 0.15 is about -1.9. We negate it to get 1.9. If the model was more confident (say 0.9), the NLL would be just 0.1. So NLL measures how surprised the model is by the actual data. Less surprise = lower loss = better model.</p>
</Callout>

## Neural Bigram

Now let us do the exact same thing -- predict the next character from one character -- but with a neural network instead of counting. Why? Because neural networks can be extended to handle more context, and they learn via gradients. This is the first step toward a transformer.

Instead of a count matrix, we have a *weight* matrix `W` that the network will learn. We feed in a character as a one-hot vector, multiply by `W`, and get a score for every possible next character. Then we apply `softmax` to convert scores to probabilities.

<Callout type="info" title="Why softmax specifically?">
  <p>The weight matrix produces raw scores (called &quot;logits&quot;) that can be any real number -- negative, positive, huge, tiny. But we need a valid probability distribution: all values positive, summing to 1. Softmax does this in two steps. First, it exponentiates every logit: <code>exp(x)</code> maps any real number to a positive number. Second, it divides each by the sum of all exponentiated values, forcing everything to sum to 1. The exponential has a nice property: it preserves the ordering (bigger logits get bigger probabilities) while ensuring nothing is negative. A useful concept is &quot;temperature&quot;: if you divide logits by a temperature T before softmax, T &gt; 1 makes the distribution more uniform (more random generation) and T &lt; 1 makes it more peaked (more deterministic). At T approaching 0, it becomes argmax.</p>
</Callout>

<CodeBlock title="neural_bigram.py" language="python">{`# Neural bigram model
g = torch.Generator().manual_seed(2147483647)
W = torch.randn((vocab_size, vocab_size), generator=g, requires_grad=True)

# Build training data: pairs of (input_char, target_char)
xs, ys = [], []
for w in train_words:
    chs = ['.'] + list(w) + ['.']
    for ch1, ch2 in zip(chs, chs[1:]):
        xs.append(stoi[ch1])
        ys.append(stoi[ch2])

xs = torch.tensor(xs)
ys = torch.tensor(ys)

print(f"Training examples: {len(xs)}")`}</CodeBlock>

<Callout type="tip" title="One-hot encoding">
  <p>A one-hot vector is like a boolean array where exactly one element is <code>true</code>. For character index 5 with vocab size 27, it is <code>[0,0,0,0,0,1,0,...,0]</code>. In PyTorch, <code>F.one_hot(torch.tensor(5), 27)</code> creates this. When you multiply a one-hot vector by a matrix, you just pluck out one row -- it is a lookup.</p>
</Callout>

<CodeBlock title="neural_bigram_train.py" language="python">{`# Training loop
for epoch in range(200):
    # Forward pass
    xenc = F.one_hot(xs, num_classes=vocab_size).float()  # one-hot encode inputs
    logits = xenc @ W  # predict scores for next character (@ is matrix multiply)
    counts = logits.exp()  # exponentiate to get fake "counts"
    probs = counts / counts.sum(dim=1, keepdim=True)  # normalize to probabilities

    # Loss: negative log likelihood (same metric as before)
    loss = -probs[torch.arange(len(ys)), ys].log().mean()

    # Backward pass
    W.grad = None  # zero gradients (like optimizer.zero_grad())
    loss.backward()  # compute gradients -- PyTorch does this automatically

    # Update weights
    W.data -= 50 * W.grad  # gradient descent with learning rate 50

    if epoch % 50 == 0:
        print(f"Epoch {epoch}: loss = {loss.item():.4f}")

# Final loss
print(f"Final training loss: {loss.item():.4f}")  # ~2.47`}</CodeBlock>

There is a lot happening here, so let us unpack the key ideas:

- **Forward pass**: Convert the input character to a one-hot vector, multiply by the weight matrix `W` to get "logits" (raw scores), then apply softmax (exponentiate and normalize) to get probabilities. The `@` operator is matrix multiplication -- like `numpy.dot` or manual nested loops in JS.
- **Loss**: We grab the probability the model assigned to the *correct* next character, take the log, negate it, and average. This is cross-entropy loss. Why is this the same as the NLL we computed for the bigram model? Cross-entropy between two distributions p (the true one) and q (our model) is defined as -sum(p * log(q)). When p is a one-hot vector (the true next character), the sum collapses to just -log(q_correct) -- the negative log probability of the correct class. So cross-entropy loss and NLL are the same thing when the target is a single correct answer.
- **Backward pass**: `loss.backward()` is where the magic happens. PyTorch traces every operation that produced `loss` and computes the gradient of `loss` with respect to `W`. This is called *autograd* -- automatic differentiation. But why does this work? The gradient of the loss with respect to each weight tells you: "if I increase this weight by a tiny amount, how much does the loss change?" The gradient points in the direction of steepest increase. So we go the opposite direction to decrease the loss. This is the chain rule from calculus, applied automatically through every operation (matrix multiply, softmax, log) back to the original weights. That automatic application of the chain rule is called backpropagation, and `loss.backward()` does it all for you.
- **Update**: We nudge `W` in the direction that reduces the loss. The learning rate (50 here, which is unusually large because our model is so simple) controls the step size. You can think of the loss as a hilly landscape where we are trying to find the lowest valley. The gradient tells us which direction is "downhill," and the learning rate determines how big of a step we take.

After training, the weight matrix `W` essentially converges to the same thing as our count-based probability matrix `P`. The neural network rediscovered counting through gradient descent. Now let us sample from it:

<CodeBlock title="neural_bigram_sample.py" language="python">{`# Sample from the neural bigram
g = torch.Generator().manual_seed(2147483647)

for _ in range(10):
    out = []
    ix = 0
    while True:
        xenc = F.one_hot(torch.tensor([ix]), num_classes=vocab_size).float()
        logits = xenc @ W
        probs = F.softmax(logits, dim=1)
        ix = torch.multinomial(probs, num_samples=1, replacement=True, generator=g).item()
        if ix == 0:
            break
        out.append(itos[ix])
    print(''.join(out))`}</CodeBlock>

That is expected -- same model, same result. But now we have the machinery to go further. We can make the network deeper, give it more context, and it will learn things that counting never could.

<Callout type="info" title="Why bother with a neural network if counting works?">
  <p>Counting only works for bigrams. If we wanted to condition on 3 characters, we would need a 27x27x27 tensor. For 10 characters of context, that is 27^10 entries -- impossible. Neural networks scale because they learn compressed representations instead of storing every combination.</p>
</Callout>

## MLP Language Model

Time for a real upgrade. Instead of looking at just one character, we will look at the last 3 characters (a "context window") and use a multi-layer perceptron (MLP) to predict the next one. This is the architecture from Bengio et al. (2003) -- a classic.

The key new idea is **embeddings**. Instead of a one-hot vector (27 dimensions, mostly zeros), we represent each character as a small dense vector (say, 10 dimensions). The network learns these embeddings during training. Characters that behave similarly (like vowels) end up with similar embeddings.

<Callout type="info" title="Why embeddings instead of one-hot vectors?">
  <p>One-hot encoding has a fundamental problem: it treats every character as equally different from every other character. The distance between &apos;a&apos; and &apos;e&apos; is the same as between &apos;a&apos; and &apos;z&apos;. But we know that vowels behave similarly, and certain consonant clusters appear together. One-hot vectors live in a 27-dimensional space where all characters are orthogonal -- they share zero information. This is the curse of dimensionality: as the vocabulary grows, the space becomes increasingly sparse and the model cannot generalize. Embeddings solve this by projecting characters into a much smaller space (10 dimensions here) where the model can learn that similar characters are nearby. After training, you would find that vowels cluster together and consonants form their own groups. This compressed, learned representation is one of the most powerful ideas in deep learning.</p>
</Callout>

<MLPForwardPass />

The flow is: look up embeddings for the last 3 characters, concatenate them into one long vector, pass through a hidden layer with `tanh` activation, then project to vocabulary-size output.

<CodeBlock title="mlp_data.py" language="python">{`# Build dataset with context window
block_size = 3  # how many characters of context

def build_dataset(words):
    X, Y = [], []
    for w in words:
        context = [0] * block_size  # start with all '.' tokens
        for ch in w + '.':
            ix = stoi[ch]
            X.append(context)
            Y.append(ix)
            context = context[1:] + [ix]  # slide the window forward
    X = torch.tensor(X)
    Y = torch.tensor(Y)
    return X, Y

X_train, Y_train = build_dataset(train_words)
X_val, Y_val = build_dataset(val_words)

print(f"Training set: {X_train.shape}")  # (N, 3) -- N examples, 3 chars of context
print(f"Example: {X_train[0]} -> {Y_train[0]}")  # [0, 0, 0] -> 5 (e.g., '...' -> 'e')`}</CodeBlock>

Now the model definition. In PyTorch, tensors with `requires_grad=True` are trainable parameters. We group them in a list for convenience.

<CodeBlock title="mlp_model.py" language="python">{`# MLP model
n_embd = 10   # embedding dimension
n_hidden = 200  # hidden layer size

g = torch.Generator().manual_seed(2147483647)

C = torch.randn((vocab_size, n_embd), generator=g)  # embedding lookup table
W1 = torch.randn((block_size * n_embd, n_hidden), generator=g) * 0.2  # hidden layer
b1 = torch.randn(n_hidden, generator=g) * 0.01
W2 = torch.randn((n_hidden, vocab_size), generator=g) * 0.01  # output layer
b2 = torch.randn(vocab_size, generator=g) * 0.01

parameters = [C, W1, b1, W2, b2]
for p in parameters:
    p.requires_grad = True

print(f"Total parameters: {sum(p.nelement() for p in parameters)}")  # ~6,000`}</CodeBlock>

<Callout type="tip" title="Weight initialization matters">
  <p>We multiply initial weights by small numbers (0.2, 0.01) to prevent the outputs from being too extreme at the start. If logits are huge, softmax saturates and gradients vanish. This is a common source of bugs in ML. Think of it like setting reasonable default values in a React component.</p>
</Callout>

<CodeBlock title="mlp_train.py" language="python">{`# Training loop with mini-batches
for epoch in range(20000):
    # Mini-batch: randomly sample 64 examples
    ix = torch.randint(0, X_train.shape[0], (64,), generator=g)
    Xb, Yb = X_train[ix], Y_train[ix]

    # Forward pass
    emb = C[Xb]                           # (64, 3, 10) -- look up embeddings
    h = emb.view(-1, block_size * n_embd)  # (64, 30) -- flatten context
    h = torch.tanh(h @ W1 + b1)           # (64, 200) -- hidden layer with tanh
    logits = h @ W2 + b2                   # (64, 27) -- output scores
    loss = F.cross_entropy(logits, Yb)     # compute loss

    # Backward pass
    for p in parameters:
        p.grad = None
    loss.backward()

    # Update: decay learning rate over time
    lr = 0.1 if epoch < 10000 else 0.01
    for p in parameters:
        p.data -= lr * p.grad

    if epoch % 5000 == 0:
        print(f"Step {epoch}: loss = {loss.item():.4f}")

# Evaluate on validation set
emb = C[X_val]
h = torch.tanh(emb.view(-1, block_size * n_embd) @ W1 + b1)
logits = h @ W2 + b2
val_loss = F.cross_entropy(logits, Y_val)
print(f"Validation loss: {val_loss.item():.4f}")  # ~2.1`}</CodeBlock>

Some important new concepts:

- **Mini-batches**: Instead of using all training data each step, we randomly sample 64 examples. This is not just about speed, though it is faster. The noise introduced by random sampling is actually beneficial. Computing the gradient on the full dataset gives you the "true" gradient, but following it precisely tends to converge to sharp, narrow minima that do not generalize well. The noisy gradient from a mini-batch helps the optimizer escape these sharp minima and find flatter regions of the loss landscape -- and flatter minima correspond to solutions that generalize better to unseen data. This is stochastic gradient descent (SGD), and the stochasticity is a feature, not a bug.
- **Embedding lookup**: `C[Xb]` uses the character indices to index into the embedding table. If `Xb[0]` is `[5, 13, 13]`, we get rows 5, 13, and 13 from `C`. This is more efficient than one-hot multiplication and does the same thing.
- **`view(-1, 30)`**: Reshapes the tensor. The `-1` means "figure out this dimension automatically." It is like `array.flat()` but for multi-dimensional tensors with explicit target shape.
- **`tanh`**: An activation function that squishes values between -1 and 1. It introduces *non-linearity* -- without it, stacking linear layers would just be one big linear layer (matrix times matrix is still a matrix). This point deserves emphasis: if you stack two linear layers without a nonlinearity, the result is `(x @ W1) @ W2 = x @ (W1 @ W2)` -- just a single matrix multiply. No matter how many linear layers you stack, you get a linear function. The nonlinearity is what gives neural networks their power. The universal approximation theorem says that a network with even one hidden layer and a nonlinear activation can approximate *any* continuous function, given enough neurons. Tanh, ReLU, sigmoid -- the specific choice matters for training dynamics, but the key insight is that you need *something* nonlinear.

<Callout type="info" title="Why learning rate decay?">
  <p>Notice how the learning rate drops from 0.1 to 0.01 halfway through training. Early on, the model is far from any good solution, so you want large steps to cover ground quickly. But as training progresses and the model gets close to a minimum, large steps cause it to overshoot and oscillate around the optimum without ever settling in. Reducing the learning rate lets the model fine-tune its weights with smaller, more precise adjustments. Without decay, you often see the loss bounce around a good value without actually reaching it. More sophisticated optimizers like AdamW (which we will use later) handle some of this automatically, but explicit scheduling still helps.</p>
</Callout>

The model learned that consonant clusters matter, that names have rhythm, and that certain endings (`-lyn`, `-ie`, `-on`) are common. All from 3 characters of context and 6,000 parameters.

## Self-Attention

Here is where things get interesting. The MLP treats its context window as a fixed-size concatenated vector. Position 1, 2, and 3 each have their own dedicated weights. But what if the model could *dynamically decide* which positions to pay attention to?

Consider the fundamental limitation of the MLP approach. With `block_size=3`, the model can only see 3 characters of context. A name like `"christopher"` has 11 characters -- the model predicting the final `"r"` has no idea it started with `"ch"`. You could increase the block size, but then the number of parameters grows linearly with context length, every position has its own dedicated weights, and the model cannot learn that a pattern at position 2 and the same pattern at position 7 are the same thing. The MLP is fundamentally position-rigid.

That is the core idea of self-attention. Each position in the sequence asks: "Which other positions have information I need?" It then creates a weighted average of information from those positions. Any position can talk to any other position, and the communication pattern is learned from data, not hardwired.

Let us build the intuition with three concepts:

- **Query (Q)**: "What am I looking for?" -- Each position creates a query vector.
- **Key (K)**: "What do I contain?" -- Each position also creates a key vector.
- **Value (V)**: "What will I share if someone attends to me?" -- The actual information to pass along.

### A library analogy

Imagine you walk into a library looking for information. You have a **question in your head** (your query). Every book on the shelves has a **title and topic on the spine** (its key) -- a short advertisement of what it contains. You scan the spines, and the books whose topics best match your question are the ones you pull off the shelf. But the title is not what you actually read -- once you pull a book down, you read the **contents inside** (its value).

That is exactly what self-attention does at every position in the sequence:

1. Each position forms a query: "What kind of context would help me predict the next character?"
2. Every position (including itself) advertises a key: "Here is what I am, in a form designed to be matched against queries."
3. The query is compared against every key (via dot product). High match = high attention.
4. Each position also has a value: the actual information it will contribute. We take a weighted average of all values, weighted by how well each key matched our query.

The crucial design choice: a position's key (how it advertises itself) is *different* from its value (what it actually contributes). The library's spine summary is not the same as the book's contents. This separation lets the model learn very different things for the two roles.

### A concrete example: predicting the next character in "alexandr"

Suppose the model has read `a-l-e-x-a-n-d-r` and is about to predict the next character. The position holding `r` (position 8) needs to decide what comes next. A human seeing this would think `"a"` -- because we have seen `"alex"` and `"andr"` before, and the `"andr-"` pattern often ends in `"-a"` (alexandra, sandra, cassandra).

How does self-attention figure this out? The `r` at position 8 builds a **query** vector. Conceptually, this query might encode something like: *"I am a consonant following 'and'. I want to know about the broader name structure -- specifically, what vowels appeared earlier and whether this looks like a feminine ending pattern."* (The model never literally writes that down -- it learns a high-dimensional vector that, through training, has come to mean something like that.)

Now every earlier position (1 through 8) has already built a **key**. Each key is that position's "spine summary":

- Position 1 (`a`): "I am a name-initial vowel"
- Position 2 (`l`): "I am an early consonant after a vowel"
- Position 3 (`e`): "I am a vowel in position 3"
- Position 4 (`x`): "I am the rare consonant 'x'"
- Position 5 (`a`): "I am a vowel after 'x'"
- ...and so on

The query from position 8 is dot-producted with every one of these keys. Suppose the dot products come out to `[0.2, 0.1, 0.3, 2.1, 0.4, 1.8, 1.5, 0.9]`. After softmax, position 8 ends up paying most of its attention to positions 4 (`x`), 6 (`n`), and 7 (`d`) -- the distinctive consonants that define this name's structure.

Then we use these attention weights to combine the **values**. The value at position 4 is not the spine "I am the letter x" -- it is the rich information *the model actually wants to pass forward*, which might be something like: *"a rare consonant has appeared, biasing toward names like alexander/alexandra."* Position 8 sums up all these values, weighted by attention, and uses the result to predict the next character.

<Callout type="info" title="Why separate Q, K, and V? Why not just use embeddings directly?">
  <p>You might wonder: why not just take the dot product of the raw embeddings to compute attention? The separation into queries, keys, and values lets the model learn different roles for the same position. The query projection learns &quot;what am I looking for?&quot; while the key projection learns &quot;what do I advertise to others?&quot; and the value projection learns &quot;what information do I actually share?&quot; These can be very different. The letter &apos;a&apos; at position 5 might be looking for a preceding consonant (its query) while advertising that it is a vowel (its key) and sharing its specific identity (its value). Without this separation, you would be forced to use the same representation for all three roles, which severely limits what the model can learn.</p>
</Callout>

### From intuition to math

Each of Q, K, and V is produced by a learned linear projection of the position's embedding. If `x` is the embedding (shape `C`), then `q = x @ W_q`, `k = x @ W_k`, `v = x @ W_v`, where `W_q`, `W_k`, `W_v` are learnable matrices of shape `(C, head_size)`. Three matrices, three roles -- and the model learns all three from data via gradient descent.

Stacking all `T` positions of the sequence into a single matrix `X` (shape `T × C`), we get all the queries, keys, and values for the whole sequence in one matmul each: `Q = X @ W_q`, `K = X @ W_k`, `V = X @ W_v`. The visualizer below walks through this computation cell-by-cell -- toggle Q, K, V to see the *same* input X being projected three different ways, and drag the sliders to see how the shapes change with sequence length, embedding dim, and head size.

<QKVMatrixViz />

Attention scores are computed by taking the dot product of queries with keys. If a query and key point in similar directions (high dot product), that position gets more attention. We then softmax these scores into a probability distribution and use them as weights for a sum over values. In one line: `output = softmax(Q @ K.T / sqrt(d)) @ V`.

A few things to keep in mind as you read the code below:

- The same input `x` is fed into all three projections. Q, K, V are *different views* of the same sequence.
- `Q @ K.T` produces a `(T, T)` matrix -- one attention score per pair of positions. Row `i` tells you how much position `i` attends to every other position.
- After the softmax, each row sums to 1 -- so each position's output is a true weighted average of values.
- The causal mask (next callout) zeros out the upper triangle so positions cannot peek into the future.

<CodeBlock title="attention.py" language="python">{`# Single-head self-attention from scratch
import torch
import torch.nn as nn

torch.manual_seed(42)

# Parameters
B, T, C = 4, 8, 32  # batch, time (sequence length), channels (embedding dim)
n_head_size = 16

# Simulated input: random embeddings for a sequence of 8 characters
x = torch.randn(B, T, C)

# Attention projections (these are learned)
key   = nn.Linear(C, n_head_size, bias=False)
query = nn.Linear(C, n_head_size, bias=False)
value = nn.Linear(C, n_head_size, bias=False)

k = key(x)    # (B, T, 16)
q = query(x)  # (B, T, 16)
v = value(x)  # (B, T, 16)

# Compute attention scores: how much does each position attend to each other?
weights = q @ k.transpose(-2, -1)  # (B, T, T) -- dot product of all query-key pairs
weights = weights * (n_head_size ** -0.5)  # scale down to prevent softmax saturation

# Causal mask: prevent looking at future positions
tril = torch.tril(torch.ones(T, T))
weights = weights.masked_fill(tril == 0, float('-inf'))

# Softmax to get attention probabilities
weights = F.softmax(weights, dim=-1)  # (B, T, T) -- each row sums to 1

# Weighted sum of values
out = weights @ v  # (B, T, 16)

print(f"Attention weights shape: {weights.shape}")
print(f"Output shape: {out.shape}")
print(f"Attention pattern for first sequence, last position:")
print(weights[0, -1].data)`}</CodeBlock>

<AttentionMatrix />

The diagram shows the causal mask in action. Each row represents a position, and the cells show how much it attends to other positions. The `--` cells are masked out -- the model cannot look into the future.

<Callout type="info" title="Why causal masking?">
  <p>The mask exists because of how we generate text. At generation time, when predicting character 6, we only have characters 1 through 5 -- character 7 does not exist yet. So during training, we must enforce the same constraint: each position can only attend to positions before it (and itself). If we let position 5 see position 6 during training, the model would learn to &quot;cheat&quot; by just copying the answer from the future. The causal mask (setting future positions to negative infinity before softmax) ensures the model learns to predict using only past context -- which is all it will have when generating new names.</p>
</Callout>

<Callout type="info" title="Why scale by sqrt(d_k)?">
  <p>The scaling factor <code>n_head_size ** -0.5</code> (that is, dividing by the square root of the head dimension) comes directly from the &quot;Attention is All You Need&quot; paper. Here is why it matters: the dot product of two random vectors with dimension d has a variance that grows proportionally to d. So as the head size increases, the dot products get larger in magnitude. Large dot products push the softmax into saturated regions where the output is nearly one-hot (almost all the attention goes to one position). In those saturated regions, gradients are near zero, so the model stops learning. Dividing by sqrt(d) keeps the variance of the dot products roughly constant regardless of dimension, keeping softmax in a well-behaved range where gradients flow and the model can actually train.</p>
</Callout>

Now let us extend to **multi-head attention**. Instead of one set of Q, K, V projections, we use several (say, 4) and concatenate their outputs. Each head can learn to attend to different things -- one might focus on recent characters, another on vowel patterns.

Why use multiple heads instead of one big attention? A single attention head computes one set of attention weights -- one pattern of "who attends to whom." But language has many simultaneous relationships. One head might learn "attend to the previous vowel," another might learn "attend to the first character of the name," and a third might learn "attend to the most recent consonant cluster." By running multiple heads in parallel and concatenating their outputs, the model can capture all these different types of relationships simultaneously. Each head operates in a smaller subspace (head_size = n_embd / n_heads), so the total computation is the same as one big head, but the expressiveness is much greater.

<CodeBlock title="multihead.py" language="python">{`class Head(nn.Module):
    """One head of self-attention"""
    def __init__(self, head_size):
        super().__init__()
        self.key = nn.Linear(n_embd, head_size, bias=False)
        self.query = nn.Linear(n_embd, head_size, bias=False)
        self.value = nn.Linear(n_embd, head_size, bias=False)
        self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size)))

    def forward(self, x):
        B, T, C = x.shape
        k = self.key(x)
        q = self.query(x)
        v = self.value(x)

        weights = q @ k.transpose(-2, -1) * (C ** -0.5)
        weights = weights.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
        weights = F.softmax(weights, dim=-1)
        return weights @ v

class MultiHeadAttention(nn.Module):
    """Multiple heads of self-attention in parallel"""
    def __init__(self, num_heads, head_size):
        super().__init__()
        self.heads = nn.ModuleList([Head(head_size) for _ in range(num_heads)])
        self.proj = nn.Linear(num_heads * head_size, n_embd)

    def forward(self, x):
        out = torch.cat([h(x) for h in self.heads], dim=-1)
        return self.proj(out)`}</CodeBlock>

<Callout type="info" title="nn.Module explained">
  <p>If you are coming from JS, think of <code>nn.Module</code> as a base class (like extending <code>React.Component</code>). You define parameters in <code>__init__</code> and the computation in <code>forward</code>. PyTorch automatically collects all parameters for gradient updates. The <code>register_buffer</code> method stores tensors that are not trainable (like the mask).</p>
</Callout>

### Hyperparameters

A few numbers drive the size and shape of every tensor in the model:

<CodeBlock title="hparams.py" language="python">{`block_size = 16   # context length: characters the model can see at once
n_embd = 64       # embedding dimension: each char becomes a 64-dim vector
n_head = 4        # number of attention heads (n_embd must be divisible by n_head)

dropout = 0.2     # zero-out probability during training
learning_rate = 3e-4

batch_size = 64

### Single-head attention (with dropout)

This is the same `Head` class we built in the self-attention section, with one new piece: a `nn.Dropout` layer applied to the attention weights. Dropout randomly zeros some entries during training so the model cannot lean on any single connection (more on dropout below). The shapes are unchanged: input `(B, T, n_embd)` → output `(B, T, head_size)`.

<CodeBlock title="head.py" language="python">{`class Head(nn.Module):
    """Single head of self-attention"""
    def __init__(self, head_size):
        super().__init__()
        self.key = nn.Linear(n_embd, head_size, bias=False)
        self.query = nn.Linear(n_embd, head_size, bias=False)
        self.value = nn.Linear(n_embd, head_size, bias=False)
        self.register_buffer('tril', torch.tril(torch.ones(block_size, block_size)))
        self.dropout = nn.Dropout(dropout)

    def forward(self, x):
        B, T, C = x.shape
        k = self.key(x)
        q = self.query(x)
        v = self.value(x)
        weights = q @ k.transpose(-2, -1) * (C ** -0.5)
        weights = weights.masked_fill(self.tril[:T, :T] == 0, float('-inf'))
        weights = F.softmax(weights, dim=-1)
        weights = self.dropout(weights)
        return weights @ v
`}</CodeBlock>

### Multi-head attention

Run `n_head` heads in parallel, concatenate their outputs along the feature axis, and project back to `n_embd` with a learned linear layer. Each head sees `(B, T, n_embd)` going in and produces `(B, T, head_size)` where `head_size = n_embd / n_head`. After concatenation across heads, the shape is back to `(B, T, n_embd)` -- so the block can be stacked freely.

<CodeBlock title="multihead.py" language="python">{`class MultiHeadAttention(nn.Module):
    def __init__(self, num_heads, head_size):
        super().__init__()
        self.heads = nn.ModuleList([Head(head_size) for _ in range(num_heads)])
        self.proj = nn.Linear(num_heads * head_size, n_embd)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x):
        out = torch.cat([h(x) for h in self.heads], dim=-1)
        return self.dropout(self.proj(out))
`}</CodeBlock>

### Feed-forward network

Attention is good at one thing -- mixing information across positions. But the mixing is essentially linear in the values (it is a weighted sum). The feed-forward network adds per-position nonlinear processing power. Think of attention as "gather relevant information from context" and the FFN as "process that gathered information into something useful." The 4× expansion (`n_embd → 4·n_embd → n_embd`) gives the FFN a wider intermediate layer with more capacity to learn complex transformations before projecting back down. This bottleneck-expand-compress pattern is a common design in deep learning.

<CodeBlock title="ffn.py" language="python">{`class FeedForward(nn.Module):
    """Simple feed-forward network: linear -> ReLU -> linear"""
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(n_embd, 4 * n_embd),
            nn.ReLU(),
            nn.Linear(4 * n_embd, n_embd),
            nn.Dropout(dropout),
        )

    def forward(self, x):
        return self.net(x)
`}</CodeBlock>

### Transformer block

The Block wires one attention sub-layer and one FFN sub-layer with residual (skip) connections. We use the **pre-norm** variant (the GPT-style one used in the diagram above): LayerNorm comes *before* each sub-layer, and the residual carries the unmodified input around. The two lines of `forward` are exactly what each green `+` node in the diagram represents:

```python
x = x + self.sa(self.ln1(x))    # attention sub-layer with residual
x = x + self.ffwd(self.ln2(x))  # FFN sub-layer with residual
```

**Why residual connections?** They solve the *degradation problem*. In theory, a deeper network should always be at least as good as a shallower one -- the extra layers could just learn the identity function and pass data through unchanged. But in practice, deeper networks are *harder* to train and often perform worse, because gradients vanish or explode while traveling through every layer during backpropagation. Residual connections fix this by making the identity the default: each layer only needs to learn the *delta* on top of the input. There is also a gradient perspective -- the addition operator gives the gradient a "highway" that flows directly from later layers to earlier ones without passing through any transformation that could distort it. This was the key insight of ResNet (2015), and it is why we can train networks with dozens or hundreds of layers.

<Callout type="info" title="Why layer normalization?">
  <p>As data flows through many layers of transformations, the distribution of activations can drift -- values might grow very large or very small, making training unstable. Layer normalization re-centers (subtracts mean) and re-scales (divides by standard deviation) the activations at each layer, keeping them in a well-behaved range. Why LayerNorm instead of BatchNorm (which normalizes across the batch dimension)? BatchNorm computes statistics across the batch, which means its behavior changes depending on what other examples happen to be in the batch -- problematic for sequences of varying length. LayerNorm normalizes across the feature dimension within each individual example, so it is self-contained and consistent regardless of batch composition. This makes it the standard choice for transformers.</p>
</Callout>

<CodeBlock title="block.py" language="python">{`class Block(nn.Module):
    """Transformer block: attention + FFN with pre-norm residuals"""
    def __init__(self):
        super().__init__()
        head_size = n_embd // n_head
        self.sa = MultiHeadAttention(n_head, head_size)
        self.ffwd = FeedForward()
        self.ln1 = nn.LayerNorm(n_embd)
        self.ln2 = nn.LayerNorm(n_embd)

    def forward(self, x):
        x = x + self.sa(self.ln1(x))    # attention sub-layer with residual
        x = x + self.ffwd(self.ln2(x))  # FFN sub-layer with residual
        return x
`}</CodeBlock>

### Putting it all together: NameTransformer

The outer model wires up the input embeddings, the stack of `n_layer` Blocks, a final LayerNorm, and the output projection (`lm_head`). The forward pass is:

1. **Token embedding**: look up an `n_embd`-dim vector for each character → `(B, T, n_embd)`
2. **Position embedding**: look up an `n_embd`-dim vector for each position 0..T-1 → `(T, n_embd)`, broadcast across batch
3. **Add them**: `x = tok_emb + pos_emb` → `(B, T, n_embd)`
4. **Stacked blocks**: pass through `n_layer` Blocks (shape preserved at every step)
5. **Final LayerNorm**, then `lm_head` projects to `(B, T, vocab_size)` logits

**About positional embeddings.** Attention is *permutation-invariant* by default. Shuffle the order of the input positions and the attention mechanism produces the same output (just shuffled). Without positional information, the model literally cannot distinguish `"cat"` from `"tac"` -- it sees the same set of characters and computes the same relationships. Positional embeddings break this symmetry by giving each position its own vector that is added to the token embedding before anything else happens.

<Callout type="info" title="Yes — the positional embeddings are learned">
  <p>In our model, <code>self.position_embedding = nn.Embedding(block_size, n_embd)</code> creates a table of <code>block_size</code> learnable vectors -- one per slot in the context window. They start out random and are updated by gradient descent during training, just like any other parameter. The model figures out for itself what each position should &quot;mean&quot;: it might learn that position 0 is special (start-of-name), that adjacent positions are related, and so on.</p>
  <p>This is the GPT-style approach (&quot;learned positional embeddings&quot;). The original 2017 Transformer paper used <em>fixed sinusoidal</em> embeddings instead -- a hand-designed set of sine and cosine waves at different frequencies, with no learnable parameters. Both approaches work; learned embeddings are simpler and let the model adapt the representation to the data, but they only work for positions ≤ <code>block_size</code> (the model has never seen embeddings beyond that). Sinusoidal embeddings extrapolate to longer sequences for free, at the cost of being less flexible.</p>
</Callout>

<CodeBlock title="name_transformer.py" language="python">{`class NameTransformer(nn.Module):
    def __init__(self):
        super().__init__()
        self.token_embedding = nn.Embedding(vocab_size, n_embd)
        self.position_embedding = nn.Embedding(block_size, n_embd)
        self.blocks = nn.Sequential(*[Block() for _ in range(n_layer)])
        self.ln_f = nn.LayerNorm(n_embd)
        self.lm_head = nn.Linear(n_embd, vocab_size)

    def forward(self, idx, targets=None):
        B, T = idx.shape

        tok_emb = self.token_embedding(idx)                     # (B, T, n_embd)
        pos_emb = self.position_embedding(torch.arange(T))      # (T, n_embd)
        x = tok_emb + pos_emb                                   # combine token + position info
        x = self.blocks(x)                                      # apply transformer blocks
        x = self.ln_f(x)                                        # final layer norm
        logits = self.lm_head(x)                                # project to vocab size

        if targets is None:
            return logits, None

        B, T, C = logits.shape
        loss = F.cross_entropy(logits.view(B * T, C), targets.view(B * T))
        return logits, loss

    def generate(self, idx, max_new_tokens):
        for _ in range(max_new_tokens):
            idx_cond = idx[:, -block_size:]                     # crop to context window
            logits, _ = self(idx_cond)
            logits = logits[:, -1, :]                           # focus on last position
            probs = F.softmax(logits, dim=-1)
            idx_next = torch.multinomial(probs, num_samples=1)
            idx = torch.cat([idx, idx_next], dim=1)
        return idx
`}</CodeBlock>

<Callout type="info" title="What's dropout doing in there?">
  <p>You will notice <code>nn.Dropout</code> sprinkled through Head, MultiHeadAttention, and FeedForward. Randomly zeroing activations during training sounds destructive, but it is a powerful regularization technique. It forces the network to be robust -- no single neuron can become a critical point of failure because it might be dropped at any time. The network learns redundant representations. At test time (after <code>model.eval()</code>), dropout is disabled and all neurons are active, so you get the full model. Think of it like training a team where random members are absent each day -- the team learns to function without relying on any one person.</p>
</Callout>

## Preparing the data: how transformer input differs from the MLP

The transformer eats a very different kind of input than the MLP, and it is worth slowing down to see exactly how. Four things change:

1. **Context length.** The MLP saw 3 characters at a time (`block_size = 3`). The transformer sees 16 (`block_size = 16`).
2. **Shape and abstraction.** The MLP's input was a *flat* tensor of shape `(B, block_size · n_embd)` -- the embeddings of all 3 characters concatenated into one long vector, with no notion of "position 1 vs position 2." The transformer's input is a *sequence* tensor of shape `(B, T, n_embd)` -- the model treats the time axis as a first-class dimension and uses attention + positional embeddings to reason about ordering.
3. **Padding.** Because the transformer always expects exactly `block_size` time steps, contexts shorter than that get **left-padded** with the start token (`0`). When predicting the very first character of a name, the input is `[0, 0, …, 0]` -- 16 padding tokens -- and the target is whatever character starts the name.
4. **Targets at every position.** The MLP gets one prediction per example. The transformer makes a prediction at *every* position in the sequence in parallel, so the targets are themselves a `block_size`-long sequence (the inputs shifted by one). Padded positions get the sentinel value `-100` so cross-entropy ignores them.

The data prep walks every name and produces one training example per next-character prediction. Each example is a `block_size`-long input with a matching `block_size`-long sequence of targets — one target per position, so the transformer learns to predict at every step in parallel. Positions that fall before any real character are padded with the start token in the input and with `-100` in the targets so cross-entropy ignores them:

<CodeBlock title="transformer_data.py" language="python">{`# Build dataset for the transformer
def build_transformer_dataset(words):
    """Each example is (input_seq, target_seq) of length block_size.
    Targets at padded positions are -100 so cross_entropy ignores them."""
    X, Y = [], []
    for w in words:
        chs = [0] + [stoi[ch] for ch in w] + [0]  # wrap with start/end tokens
        for i in range(len(chs) - 1):
            start = max(0, i - block_size + 1)
            inp = chs[start:i + 1]      # context up to position i
            tgt = chs[start + 1:i + 2]  # next-char target at every position
            pad_len = block_size - len(inp)
            inp = [0] * pad_len + inp        # pad inputs with the '.' token (id 0)
            tgt = [-100] * pad_len + tgt     # -100 = ignore_index for cross_entropy
            X.append(inp)
            Y.append(tgt)
    return torch.tensor(X), torch.tensor(Y)

X_train_t, Y_train_t = build_transformer_dataset(train_words)
X_val_t, Y_val_t = build_transformer_dataset(val_words)
print(f"Transformer training examples: X={X_train_t.shape}, Y={Y_train_t.shape}")  # both (N, 16)`}</CodeBlock>

<Callout type="info" title="Why -100 in the targets matters">
  <p>If you padded the targets with <code>0</code> (the <code>.</code> token) instead of <code>-100</code>, you would be training the model to predict <code>.</code> any time it sees a sequence of <code>.</code>s. At sample time you start generation with a single <code>.</code> token — exactly the all-padding pattern — and the model would immediately predict <code>.</code> and stop. Validation loss would still look great because so many "easy" padded positions are trivial. Using <code>-100</code> tells <code>F.cross_entropy</code> to skip those positions entirely (it is the default <code>ignore_index</code>), so the loss reflects only real next-character predictions.</p>
</Callout>

<CodeBlock title="transformer_train.py" language="python">{`# Initialize and train
model = NameTransformer()
optimizer = torch.optim.AdamW(model.parameters(), lr=learning_rate)

print(f"Parameters: {sum(p.numel() for p in model.parameters()):,}")

for step in range(max_steps):
    # Sample a mini-batch
    ix = torch.randint(0, X_train_t.shape[0], (batch_size,))
    xb, yb = X_train_t[ix], Y_train_t[ix]

    # Forward + backward + update
    logits, loss = model(xb, yb)
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()

    if step % 2000 == 0:
        # Also compute validation loss
        with torch.no_grad():
            vix = torch.randint(0, X_val_t.shape[0], (batch_size * 4,))
            _, val_loss = model(X_val_t[vix], Y_val_t[vix])
        print(f"Step {step}: train loss {loss.item():.4f}, val loss {val_loss.item():.4f}")

# Final validation loss
with torch.no_grad():
    _, val_loss = model(X_val_t[:1000], Y_val_t[:1000])
print(f"Final validation loss: {val_loss.item():.4f}")  # ~2.16 with these hyperparameters`}</CodeBlock>

<Callout type="info" title="AdamW vs manual gradient descent">
  <p>For the bigram and MLP, we manually updated weights with <code>p.data -= lr * p.grad</code>. AdamW is a smarter optimizer that tracks the history of gradients and adapts the learning rate per-parameter. It is the standard choice for transformers. Think of it as the difference between hand-tuning CSS values and using a design system.</p>
</Callout>

This is the architecture that, scaled up by a factor of ten thousand, becomes GPT.
